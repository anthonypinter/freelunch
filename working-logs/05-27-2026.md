## Page-by-Page Details

### `index.html` — Landing Page
- Sticky nav with logo, "Live Dashboard" ghost button, "Sign In" primary button
- Hero with gradient background, headline, subtext, and two CTAs
- "How it works" section: three step cards (sign up → log meal → leaderboard)
- Dashboard preview CTA section
- Footer with quick links

### `dashboard.html` — Live Display
- Designed to be shown on a screen or projector — no user input
- Header: logo, total meals logged pill, live clock (updates every 30s)
- **Meal carousel:** fetches 10 most recent meals, auto-advances every 5s, shows meal photo, description, user avatar, display name, username, and time ago
- **Leaderboard:** auto-cycles through Weekly → Monthly → Semester every 10s, period badge updates with each switch
- Data auto-refreshes from Supabase every 2 minutes
- `overflow: hidden` on body — no scrolling, fills the viewport

### `login.html` — Auth Page
- Tabbed UI: Sign In / Create Account on the same page
- Sign In: email + password → `signInWithPassword` → redirect to `profile.html`
- Create Account: display name, username, email, password, confirm password → `signUp` + profile row insert → success message, switch to sign-in tab
- Auto-redirects to `profile.html` if already authenticated
- Scoped `<style>` block (does not inherit display-dashboard CSS)

### `submit.html` — Log a Meal
- Auth guard: redirects to `login.html` if not signed in
- Semester tag: auto-detects current semester by querying `semesters` table against today's date
- Photo upload: click-to-browse or drag-and-drop, inline preview, remove button
- Description textarea (max 280 chars)
- On submit: uploads photo to `meal-photos` Supabase Storage bucket, gets public URL, inserts meal row
- On success: shows confirmation, redirects to `profile.html` after 1.5s

### `profile.html` — User Profile
- Auth guard: redirects to `login.html` if not signed in
- Profile card: large avatar (click "Change photo" to upload new one → saved to `meal-photos/avatars/{user_id}`), display name, username/email
- Three stat pills: Total meals, This Week, This Month (pulled from leaderboard views)
- Meal grid: all user's meals, newest first — photo thumbnail, description, date, delete button
- Status badges on meal cards: **"Pending review"** (amber) and **"Rejected"** (red) — approved meals show no badge
- Sign out button in header

### `moderation.html` — Moderation Queue
- Admin-only: checks `is_admin` on the user's profile row, redirects non-admins to `index.html`
- Lists all `pending` meals oldest-first (fairness), each showing photo, description, submitter avatar/name/username, and time ago
- Pending count badge in the header updates live as meals are actioned
- Approve slides card out to the right; Reject slides card out to the left
- Shows "All clear ✅" when the queue is empty

## Database Schema Required

### Tables (run in Supabase SQL Editor)

```sql
-- Semesters
CREATE TABLE semesters (
  id uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  name text NOT NULL,
  start_date date NOT NULL,
  end_date date NOT NULL
);

INSERT INTO semesters (name, start_date, end_date)
VALUES ('Fall 2025', '2025-08-25', '2025-12-15');

-- Meals
CREATE TABLE meals (
  id uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id uuid REFERENCES auth.users(id) ON DELETE CASCADE,
  description text NOT NULL,
  photo_url text,
  semester_id uuid REFERENCES semesters(id),
  created_at timestamptz DEFAULT now()
);

ALTER TABLE meals ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users can insert their own meals" ON meals FOR INSERT WITH CHECK (auth.uid() = user_id);
CREATE POLICY "Users can delete their own meals" ON meals FOR DELETE USING (auth.uid() = user_id);

-- Profiles (added beyond original arch.md — needed for carousel user info + moderation)
CREATE TABLE profiles (
  id           uuid REFERENCES auth.users(id) ON DELETE CASCADE PRIMARY KEY,
  display_name text,
  username     text UNIQUE,
  avatar_url   text,
  is_admin     boolean DEFAULT false
);

ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Profiles are public"        ON profiles FOR SELECT USING (true);
CREATE POLICY "Users manage own profile"   ON profiles FOR ALL    USING (auth.uid() = id);
```

### Moderation — Additional SQL

```sql
-- Add status column to meals (run after creating the meals table)
ALTER TABLE meals ADD COLUMN status text NOT NULL DEFAULT 'pending';

-- Replace the open SELECT policy with one filtered by status
-- (owners can always see their own meals regardless of status)
CREATE POLICY "Approved meals are public" ON meals FOR SELECT
  USING (status = 'approved' OR auth.uid() = user_id);

-- Allow admins to update meal status
CREATE POLICY "Admins can update meal status" ON meals FOR UPDATE
  USING (EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND is_admin = true));
```

> To grant admin access to a user, run:
> ```sql
> UPDATE profiles SET is_admin = true WHERE username = 'yourusername';
> ```

### Leaderboard Views

```sql
CREATE VIEW leaderboard_weekly AS
SELECT user_id, COUNT(*) as meal_count
FROM meals
WHERE created_at >= now() - interval '7 days'
GROUP BY user_id ORDER BY meal_count DESC;

CREATE VIEW leaderboard_monthly AS
SELECT user_id, COUNT(*) as meal_count
FROM meals
WHERE created_at >= now() - interval '30 days'
GROUP BY user_id ORDER BY meal_count DESC;

CREATE VIEW leaderboard_semester AS
SELECT user_id, COUNT(*) as meal_count
FROM meals
WHERE created_at >= '2025-08-25'
GROUP BY user_id ORDER BY meal_count DESC;
```
